| 类别 | 技术选型 | 说明 |

| API文档 | Swagger/OpenAPI 3.0 | 接口文档自动生成 |

## 二、数据库设计

### 2.1 命名规范

- 表名：`crm_` 前缀（小写，下划线分隔）
- 字段名：小写，下划线分隔
- 主键：`id`（BIGINT，自增）
- 外键：`xxx_id`
- 创建时间：`create_time`（DATETIME）
- 更新时间：`update_time`（DATETIME）
- 逻辑删除：`deleted`（TINYINT，0-未删，1-已删）

### 2.2 核心表结构

#### 2.2.1 组织与项目

```sql
-- 项目表
CREATE TABLE crm_project (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    project_code VARCHAR(32) NOT NULL COMMENT '项目编号',
    project_name VARCHAR(128) NOT NULL COMMENT '项目名称',
    project_address VARCHAR(256) COMMENT '项目地址',
    owner_name VARCHAR(64) COMMENT '甲方名称',
    owner_contact VARCHAR(32) COMMENT '甲方联系人',
    owner_phone VARCHAR(32) COMMENT '甲方联系电话',
    total_contract_amount DECIMAL(15,2) COMMENT '合同总额',
    start_date DATE COMMENT '开工日期',
    end_date DATE COMMENT '竣工日期',
    project_status TINYINT DEFAULT 1 COMMENT '项目状态：1-筹备 2-在建 3-竣工 4-停工',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    deleted TINYINT DEFAULT 0,
    UNIQUE KEY uk_project_code (project_code)
) COMMENT '项目表';

-- 标段/单体表（一个项目可分多个标段）
CREATE TABLE crm_section (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    project_id BIGINT NOT NULL COMMENT '所属项目ID',
    section_code VARCHAR(32) NOT NULL COMMENT '标段编号',
    section_name VARCHAR(128) NOT NULL COMMENT '标段/单体名称',
    contract_amount DECIMAL(15,2) COMMENT '标段合同额',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    deleted TINYINT DEFAULT 0,
    FOREIGN KEY (project_id) REFERENCES crm_project(id)
) COMMENT '标段表';
```

## 四、核心业务逻辑

### 4.1 工资核算逻辑

```
月度工资 = 基本工资 + 加班工资 - 扣款
基本工资 = 日工资单价 × 实际出勤天数
加班工资 = 日工资单价 ÷ 8 × 加班小时数 × 加班系数

加班系数规则（可配置）：
- 工作日加班：1.5倍
- 休息日加班：2.0倍
- 法定节假日加班：3.0倍
```

### 4.2 履约指数计算

```
月度履约指数 = 当月实际完成产值 / 当月计划产值 × 100%

累计履约指数 = 累计实际完成产值 / 累计计划产值 × 100%

预警规则：
- 连续2个月月度履约指数 < 80% → 黄灯
- 连续3个月月度履约指数 < 70% → 红灯
```

### 4.3 计量台账流程

```
[草稿] → 提交 → [已提交] → 数据直接进入台账
                              ↓
                          [有异议]（如发现数据有误）
```

**说明**：计量数据提交后直接进入台账，无需审批等待。如发现数据有误，可发起异议，系统保留修改痕迹。

### 4.4 隐患整改闭环流程

```
[待整改] → 下发整改通知 → [整改中] → 提交整改 → [待验收] → 验收合格 → [已闭环]
                                        ↓
                                  验收不合格
                                        ↓
                                   返回整改
```

## 五、安全设计

### 5.1 认证与授权

- 用户密码使用 BCrypt 加密存储
- 使用 JWT 实现无状态认证，Token 有效期24小时
- 支持 refresh_token 续期
- 接口级权限控制，基于RBAC模型

### 5.2 数据安全

- 身份证号、银行卡号等敏感字段 AES 加密存储
- 操作日志记录所有变更操作
- 敏感操作（如删除）需二次确认

### 5.3 文件安全

- 上传文件类型白名单校验（图片：jpg/png/pdf，限制大小10MB）
- 文件存储使用 MinIO 私有对象存储，不直接暴露文件URL
- 文件访问需通过后端接口，带权限校验

## 十、台账管理核心原则

1. **数据即时生效**：所有录入数据提交后直接进入台账，无需审批等待
2. **修改留痕**：数据修改自动记录修改前值、修改后值、修改时间、修改人
3. **时间戳精确**：关键数据（计量、结算、变更）精确到分钟
4. **导出对标**：支持按项目、按时间范围导出台账Excel，格式与传统Excel台账一致
5. **多人协作**：同一数据可多人同时录入，系统自动合并或提示冲突
